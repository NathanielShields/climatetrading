Architecture

A staged data pipeline that turns raw geophysical observations into a single drought-confirmation flag, then translates that flag into a deterministic options trade and a fully attributed P&L. Click any node for module details.

End-to-end pipeline

Module map

data_ingest/

Pluggable adapters per source. Each returns a DataFrame with two timestamps: observation_ts (when the world was in this state) and available_ts (when the trader could see it). The backtest engine filters by the latter — strict anti-lookahead.

  • open_meteo_adapter — ERA5 weather + soil
  • earthdata_adapter — SMAP + MODIS/VIIRS NDVI
  • usda_nass_adapter — Crop Progress
  • yfinance_adapter — ZC=F continuous futures
  • iv_reconstruction — YZ/GK HV → calibrated ATM IV
  • real_data — orchestrates all of the above

forecasts/

A common ForecastProvider interface returning issuance time, available time, horizon, region, and forward variables (rain probability, max temp, expected precip).

  • open_meteo_provider — ERA5 hindcast as forecast
  • graphcast_provider — four-tier (docker → in-proc → file → mock)
  • earth2_provider — four-tier (docker → in-proc → file → mock)
  • _docker_runner — shared subprocess glue for the GPU container
  • mock_provider — deterministic synthetic, used in tests

backtest/

Daily simulator. Composes signals, opens/manages positions through the MarginAccount, marks the book with Black-76, applies tick-based bid-ask + commissions, and decomposes daily P&L into futures / vol / theta / cross-gamma via four full re-pricing shocks.

  • engine.py — daily loop & state machine
  • strategy.py — entry/exit rules & signal fusion
  • execution.py — bid-ask spread + commissions
  • margin.py — capital + utilisation tracking
  • attribution.py — exact P&L decomposition

market/ & reports/

Pricing primitives and output rendering.

  • market/option_pricing — Black-76 + Greeks
  • market/straddle — ATM straddle helper
  • market/contract_roll — H/K/N/U/Z calendar, last-Friday expiry
  • reports/markdown_report — human-readable summary
  • reports/plots — matplotlib P&L + attribution figures

GPU inference layer (new)

A second-process container (ag-vol-arb/earth2:latest) hosts the heavyweight ML weather models. The Python providers shell out to it via scripts/run_weather_docker.py one, mounting a host cache so a re-run for the same issue date is free. This isolates the JAX/PyTorch/CUDA stack from the host Python env — the backtest itself stays pure-CPU and importable on any OS.

Anti-lookahead invariant

The rule: on simulated date D, the engine may only consume rows where available_ts ≤ D. Each adapter is responsible for stamping both timestamps honestly — e.g. USDA Crop Progress for week ending Sunday is stamped as available_ts = Monday 4 PM ET.
Why two timestamps instead of one?

Many ag/weather datasets describe the world at time T but are not published until T + lag. Examples: USDA NASS publishes Sunday's Crop Progress on Monday; MODIS NDVI 16-day composites are available ~1 week after the composite mid-date; SMAP has a 1–3 day latency. Using only an "observation time" would silently leak the future into your signal during backtests. The pair (observation_ts, available_ts) makes the lag explicit and the filter trivial.

What about CME settle prices?

Daily settle posts ~3:30 PM CT. The engine treats settle as available same day for purposes of marking the book, but trade decisions for date D+1 are made using signals that were available before D+1 open. In a higher-fidelity build you'd want explicit intraday timestamps; the current MVP collapses everything to daily bars.
